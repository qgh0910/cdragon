# 通用开发提示词模板

> 迁移说明：本文件保留为历史汇总模板。长期行为约束已合并到项目根目录 `AGENTS.md`；可复用阶段模板已拆分到 `my_docs/prompts/`。后续发起任务时，推荐只写“引用模板 + 本次目标 + 阶段边界”。

适用场景：使用 Codex / ChatGPT / 其他 coding agent 开发新功能、修复复杂 bug、重构模块或做安全敏感改动时，先把需求、方案、评审、实施和验收拆开，避免 agent 直接跳进代码或扩大修改范围。

推荐核心流程：

```text
需求说明 -> 技术方案 -> 设计评审 -> 最终技术方案 -> TDD实施计划 -> 分步实施 -> 验收
```

其中：

- `需求说明` 负责明确要做什么、不做什么、成功标准是什么。
- `技术方案` 负责基于当前代码库设计怎么做。
- `设计评审` 负责找风险、漏洞、遗漏和过度设计。
- `最终技术方案` 负责吸收评审结论，成为实施的唯一依据。
- `TDD实施计划` 负责把开发拆成可逐步执行的小步骤。
- `分步实施` 负责每次只执行一个步骤，先测试、再实现、再验证。
- `验收` 负责对照需求和最终方案确认功能是否真正完成。

## 0. 使用前替换项

每次使用前先替换这些占位符：

```text
{功能名称}
{功能描述}
{日期}
{项目文档目录}
{进度文件}
{第N步}
```

建议默认值：

```text
{项目文档目录}=my_docs
{进度文件}=my_docs/PROJECT_PROGRESS.md
{日期}=YYYY-MM-DD
```

建议文档命名：

```text
my_docs/{日期}-{功能名称}-需求说明.md
my_docs/{日期}-{功能名称}-技术方案.md
my_docs/{日期}-{功能名称}-设计评审报告.md
my_docs/{日期}-{功能名称}-最终技术方案.md
my_docs/{日期}-{功能名称}-TDD实施计划.md
```

## 1. 通用约束提示词

```text
你是一个谨慎的高级全栈工程师。
请基于当前代码库进行分析和修改，不要臆测不存在的文件、接口或实现。

通用要求：
1. 除非我明确要求进入实施阶段，否则不要修改代码。
1. 除非用户明确要求进入实施阶段，否则不要修改代码。
2. 不要回退、覆盖或删除用户已有改动。
3. 不要提前实现后续步骤。
4. 如果当前代码库与文档描述不一致，以当前代码库为准，并在输出中说明差异。
5. 涉及权限、认证、密码、用户数据、支付、数据库迁移、批量操作时，必须额外说明安全风险和边界条件。
6. 所有结论尽量基于实际文件、实际代码和实际测试结果。
7. 输出内容要结构化，方便保存到项目文档目录。
8. 不要生成过度复杂的方案，优先选择与当前项目风格一致、可渐进实现、可测试的方案。
9. 不要主动提交、回退、删除或大范围重构，除非用户明确要求。
10. 修改代码前，先说明将要修改的范围；实施时只改本阶段相关文件。
11. 完成后必须说明：
    - 修改了哪些文件
    - 做了哪些验证
    - 是否有未验证项
    - 是否存在遗留风险
12. 如果当前 Codex 环境安装了 Superpowers，并且本阶段适合使用对应能力，则优先使用：
    - brainstorming：用于需求拆解、范围澄清、技术方案比较；仅在方案阶段使用，不得提前写代码。
    - writing-plans：用于已有明确需求或已确认方案后的实施计划。
    - review：用于方案自查、代码审查、风险检查；代码审查优先使用 Superpowers 的 code review 相关能力。
    - TDD：用于明确进入实施阶段后的测试先行开发。
    - verification：用于实施后的验证与总结，必须有实际验证证据后再声明完成。
    但必须遵守用户指定的阶段边界；如果用户只要求计划/审查/分析，不得因为工具能力而提前修改代码或实现后续步骤。
```


## 2. 流程分级判断提示词

```text
请判断以下需求应该走哪种开发流程：

需求：
{功能描述}

请根据需求复杂度、安全风险、权限影响、数据库影响、影响范围判断：

1. 是否可以直接写小实施计划
2. 是否需要：需求说明 -> 技术方案 -> TDD实施计划
3. 是否需要完整流程：需求说明 -> 技术方案 -> 设计评审 -> 最终技术方案 -> TDD实施计划 -> 分步实施 -> 验收

判断标准：
- 小 bug、文案、样式、小范围前端交互：可以直接写小实施计划。
- 普通功能、影响多个文件、有一定业务规则：需求说明 -> 技术方案 -> TDD实施计划。
- 涉及认证、权限、密码、用户状态、数据库迁移、支付、数据安全、批量操作、重要业务流程：完整流程。
- 如果已有父级最终方案覆盖本功能，可以降级为：需求说明 -> 技术方案 -> TDD实施计划，但技术方案中必须包含安全风险与决策记录。

请输出：
1. 推荐流程
2. 判断理由
3. 是否需要设计评审
4. 是否需要最终技术方案
5. 推荐文档命名
6. 下一步应该使用哪个提示词
```

```text
功能描述：

要求：

4. 输出文件建议命名为：
   {项目文档目录}/{日期}-{功能名称}-需求说明.md
```

## 4. 技术方案提示词

```text
请使用 Superpowers 的 planning 能力，基于以下需求说明生成技术方案文档。
本阶段只生成技术方案，不修改代码。

需求说明文件：
{项目文档目录}/{日期}-{功能名称}-需求说明.md

要求：
1. 先阅读：
   - {进度文件}
   - 需求说明文件
   - 当前代码库中与该功能相关的模型、路由、服务、权限中间件、测试文件
   - 如有历史设计方案，也一并阅读

2. 技术方案必须基于当前代码库，不要设计当前项目明显不需要的复杂架构。

3. 技术方案必须包含：
   - 当前代码现状
   - 目标架构
   - 涉及模块
   - 数据模型设计
   - API / 路由设计
   - 权限校验设计
   - 状态流转设计
   - 安全设计
   - 错误处理设计
   - 测试策略
   - 数据迁移方案，如不需要迁移也要说明
   - 向后兼容影响
   - 风险点
   - 不采用的方案及原因
   - 实施顺序建议

4. 对于权限、认证、密码、用户状态相关功能，必须额外回答：
   - 谁可以操作？
   - 谁不可以操作？
   - 是否允许用户操作自己？
   - 是否存在最后一个管理员、最后一个所有者等保护规则？
   - 状态变更后登录、会话、缓存、令牌如何处理？
   - 接口是否会返回敏感字段？
   - 是否需要操作日志？
   - 测试如何覆盖越权访问？

5. 输出文件建议命名为：
   {项目文档目录}/{日期}-{功能名称}-技术方案.md

请直接输出完整 Markdown 文档内容。
```

## 5. 设计评审提示词

```text
请使用 Superpowers 的 review / code-review 能力，审计以下技术方案。
本阶段只做设计评审，不修改代码，不生成最终方案。

技术方案文件：
{项目文档目录}/{日期}-{功能名称}-技术方案.md

关联需求文件：
{项目文档目录}/{日期}-{功能名称}-需求说明.md

要求：
1. 先阅读：
   - {进度文件}
   - 需求说明文件
   - 技术方案文件
   - 当前代码库中相关实现
   - 相关测试文件

2. 请从以下角度审计：
   - 需求是否被完整覆盖
   - 是否存在技术风险
   - 是否存在安全风险
   - 是否存在权限绕过风险
   - 是否遗漏边界场景
   - 是否遗漏测试场景
   - 是否与当前代码结构冲突
   - 是否与已有文档或进度记录冲突
   - 是否存在过度设计
   - 是否有更小、更稳妥的实现路径
   - 是否有数据库迁移风险
   - 是否有向后兼容风险
   - 是否有用户数据泄露风险
   - 是否有日志泄露敏感信息风险
   - 是否有并发或状态一致性问题

3. 风险级别请使用：
   - P0：必须修复，否则不能进入开发
   - P1：强烈建议修复
   - P2：可以后续优化
   - P3：仅作为提醒

4. 输出格式：
   - 总体结论
   - 必须修改的问题
   - 建议修改的问题
   - 可以接受的风险
   - 遗留问题
   - 对技术方案的具体修改建议
   - 是否建议进入最终方案阶段

5. 输出文件建议命名为：
   {项目文档目录}/{日期}-{功能名称}-设计评审报告.md

请直接输出完整 Markdown 评审报告。
```

## 7. TDD 实施计划提示词

```text
请使用 Superpowers 的 TDD planning 能力，基于最终技术方案生成可逐步执行的 TDD实施计划。
本阶段只生成实施计划，不修改代码。

最终技术方案文件：
{项目文档目录}/{日期}-{功能名称}-最终技术方案.md

要求：
1. 先阅读：
   - {进度文件}
   - 最终技术方案
   - 当前代码库相关实现
   - 当前测试结构

2. 实施计划必须采用 TDD：
   - 每一步先补测试
   - 再实现最小代码
   - 再运行验证命令
   - 再记录结果

3. 每一步必须足够小，方便之后让 Codex 按第 N 步执行。

4. 每一步必须包含：
   - 步骤编号
   - 步骤目标
   - 前置条件
   - 涉及文件
   - 需要新增或修改的测试
   - 实现内容
   - 验证命令
   - 完成标准
   - 不允许提前做的事项
   - 风险与注意事项

5. 必须明确哪些步骤只写测试，哪些步骤写实现。
6. 不要把多个大功能塞进同一个步骤。
7. 不要提前实现后续步骤。
8. 最后必须包含总体验收步骤。
9. 最后必须提醒每轮实施后更新：
   {进度文件}

输出文件建议命名为：
{项目文档目录}/{日期}-{功能名称}-TDD实施计划.md

请直接输出完整 Markdown 文档内容。
```

## 8. 分步实施提示词

```text
请使用 Superpowers 的 executing-plans、test-driven-development 和 verification-before-completion 能力，按以下 TDD实施计划执行第 {第N步} 步。

实施计划文件：
{项目文档目录}/{日期}-{功能名称}-TDD实施计划.md

要求：
1. 先阅读：
   - {进度文件}
   - {项目文档目录}/{日期}-{功能名称}-最终技术方案.md
   - {项目文档目录}/{日期}-{功能名称}-TDD实施计划.md
   - 当前代码库相关文件

2. 本轮只完成第 {第N步} 步。
3. 不要提前实现后续步骤。
4. 严格采用 TDD：
   - 先补测试
   - 确认测试失败或说明为何无法单独失败
   - 再实现最小代码
   - 再运行验证命令

5. 不回退、不覆盖用户已有改动。
6. 不做与本步骤无关的重构。
7. 如果发现实施计划与当前代码冲突，先停止并说明冲突，不要强行改。
8. 如果需要调整实施计划，只能提出建议，不要擅自扩大本轮范围。
9. 完成后必须更新：
   {进度文件}

{进度文件} 需要记录：
- 当前执行步骤
- 修改文件
- 新增测试
- 实现内容
- 运行的验证命令
- 验证结果
- 遗留问题
- 下一步建议

最终输出：
- 本轮完成内容
- 修改文件列表
- 验证命令和结果
- 是否完成第 {第N步} 步
- 是否存在遗留问题
- 下一步应该执行哪一步
```

## 9. 验收提示词

```text
请使用 Superpowers 的 verification-before-completion / review 能力，对以下功能进行最终验收。

功能名称：{功能名称}

相关文档：
1. {项目文档目录}/{日期}-{功能名称}-需求说明.md
2. {项目文档目录}/{日期}-{功能名称}-最终技术方案.md
3. {项目文档目录}/{日期}-{功能名称}-TDD实施计划.md
4. {进度文件}

要求：
1. 先阅读以上文档和当前代码库。
2. 不要新增功能，不要做无关重构。
3. 本阶段只做验收、修复明显缺陷和补充必要测试。
4. 如果发现问题，请按严重程度分类：
   - 阻塞验收
   - 建议修复
   - 可后续优化

5. 验收内容必须包括：
   - 需求是否全部完成
   - 非目标是否没有被误实现
   - API 行为是否符合最终方案
   - 权限控制是否正确
   - 安全边界是否正确
   - 敏感字段是否没有泄露
   - 数据库迁移是否正确
   - 错误处理是否符合预期
   - 测试是否覆盖核心路径
   - 回归测试是否通过
   - {进度文件} 是否完整记录

6. 对于权限、认证、密码、用户管理类功能，必须额外检查：
   - 未登录访问
   - 普通用户越权访问
   - 管理员正常访问
   - 管理员操作自己的限制
   - 最后一个管理员 / 所有者保护
   - 禁用用户后的登录限制
   - 重置密码后的旧凭证失效
   - 接口响应不包含 password_hash、token、reset_token 等敏感字段

7. 请运行必要的验证命令。
8. 如果测试失败，只修复与本功能直接相关的问题。
9. 不要扩大修改范围。
10. 验收完成后更新：
    {进度文件}

最终输出：
- 验收结论：通过 / 不通过
- 已完成需求列表
- 未完成或有问题的需求列表
- 修复内容
- 修改文件
- 验证命令和结果
- 遗留问题
- 后续建议
```

## 10. Superpowers 配合方式

建议写：

```text
请使用 Superpowers 的 planning / TDD / review / verification 能力，但必须严格遵守以下阶段边界和文档命名。
```

不建议只写：

```text
请使用 Superpowers 完成这个功能。
```

原因：后者太泛，容易让 agent 自己决定流程和交付物；前者能让 Superpowers 提供方法论，同时让本模板规定项目文档结构、阶段边界和验收标准。

推荐组合：

```text
Superpowers = 方法论插件
本模板 = 项目流程规范
my_docs = 单一事实来源
PROJECT_PROGRESS.md = 执行状态记录
```

按任务类型选择流程：

```text
小任务：
请使用 Superpowers 的 TDD 能力，直接生成小实施计划并执行。

普通功能：
请使用 Superpowers 的 planning 和 TDD 能力，按：
需求说明 -> 技术方案 -> TDD实施计划 -> 分步实施
执行。

权限、登录、密码、用户管理、数据库迁移、支付、数据安全类功能：
```
